SQLAlchemy Model Plan + Alembic Migration Blueprint
Slices 14–16
Projects, Contracts, Job Costing, Budgeting, and Management Accounting v1
1. Purpose

This blueprint translates the newly locked scope extension into an implementation-safe plan for:

exact SQLAlchemy model ownership
exact package placement
migration sequencing
relationship direction
cross-module integration points
service and DTO boundaries
what Codex may implement first
what Codex must not invent

This follows the same role as the earlier Phase 1 ORM/migration blueprint: it is the bridge between locked schema direction and safe implementation.

2. Architectural fit to preserve

This new domain is not a sidecar. It must fit the existing Seeker architecture exactly.

That means:

it lives inside the same modular monolith
it uses the same feature-package discipline
company scoping remains mandatory
actual accounting truth still lives in journal_entries and journal_entry_lines
operational modules keep ownership of their own documents
reports still read from posted accounting data and approved operational facts
no UI widget may write these tables directly outside service workflows

So the correct mental model is:

new management-accounting structure over the accounting spine
not
a second accounting engine.

4. ORM conventions inherited from the existing Seeker blueprint

Use the same ORM and migration discipline already locked for Seeker:

SQLAlchemy 2.x typed declarative mapping
DeclarativeBase
Mapped[...] and mapped_column(...)
portable String, Text, Date, DateTime, Boolean, Numeric
no database-native enums
integer PKs for now
conservative mixins only where they help readability
metadata naming convention for stable Alembic output

This is non-negotiable. The new slices should look and feel like a continuation of the earlier ORM plan, not a different coding style.

7. Relationship direction to lock

This part matters a lot because bad relationships create import tangles and ORM bloat.

7.1 Within the new domain, readable bidirectional relationships are allowed

Recommended:

Contract.change_orders
Contract.projects
Project.contract
Project.jobs
Project.budget_versions
Project.commitments
ProjectJob.project
ProjectJob.parent
ProjectJob.children
ProjectBudgetVersion.project
ProjectBudgetVersion.lines
ProjectCommitment.project
ProjectCommitment.lines

These are domain-local and readable.

7.2 Across existing operational modules, keep it mostly one-way

Recommended:

source document headers and lines may have many-to-one relationships to Contract, Project, ProjectJob, and ProjectCostCode
do not create giant reverse collections like:
Project.sales_invoice_lines
Project.purchase_bill_lines
Project.inventory_document_lines
Project.journal_entry_lines
ProjectCostCode.journal_entry_lines

Reason:
Those backrefs become noisy, slow to reason about, and tightly couple the domain to every module. Cross-module reporting should come from repositories and query services, not giant ORM collection graphs.

7.3 Cross-module relationship rule

Use ORM relationships where they improve readability for normal workflows.
Use repository queries for cross-module analysis.

That is the same spirit as the existing Seeker architecture: repositories own data access, services own workflow orchestration, and models should not become business engines.

11. Source-module integration rule to lock

This is one of the most important architectural decisions.

When sales, purchases, treasury, inventory, payroll, and journals gain project fields, those modules do not transfer ownership to the new domain.

They remain the owners of:

invoice workflows
bill workflows
treasury workflows
stock workflows
payroll workflows
journal workflows

The new domain adds:

dimensional context
validation hooks
analysis and control

This preserves the existing Seeker rule that source-document posting services remain the way operational workflows create accounting impact.

12.3 Slice 16 revisions
Default recommendation

No new truth tables in slice 16.

Slice 16 should be primarily:

repositories
query services
report DTOs
UI/report pages
Optional future-only revision

Create reporting helper tables only if performance later proves it necessary, and only as controlled cache/snapshot artifacts. Do not do that in the first pass.

That stays faithful to the existing Seeker schema rule that summary/cached artifacts may exist later, but must not replace base truth.

13. What belongs in migrations versus services
13.1 Migrations own
table creation
indexes
foreign keys
unique constraints
nullable/non-nullable structure
simple static reference inserts only if truly stable
13.2 Services own
numbering orchestration
state transitions
same-company validation
contract/project compatibility checks
budget approval rules
budget-control warnings and stops
project dimension inheritance from header to line
project actual-cost derivation logic
profitability assembly logic

That is the same discipline already locked in Seeker: workflow logic belongs in services, not hidden inside Alembic or UI code.

15. What Codex must not do

Codex must not:

invent purchase-order tables in this slice
invent timesheet tables in this slice
invent WIP/revenue-recognition tables in this slice
invent a giant project_transactions table
add reverse ORM collections everywhere across modules
redesign sales, purchases, payroll, treasury, inventory, or journals
put project-profitability logic in widgets
place business logic in Alembic migrations
use SQLite-only features as the design basis
add report cache tables in the first reporting slice
merge contracts, projects, budgets, and commitments into one vague model family

These prohibitions follow directly from Seeker’s broader architecture discipline and the earlier ORM plan’s “Codex must not invent architecture” rule.

16. Acceptance criteria

This blueprint is acceptable when:

the new models map exactly to the locked schema extension
company-scoped tables consistently include company_id where required
cross-module project dimensions are added only where planned
source modules keep ownership of their workflows
actual costing is query-derived from posted source documents and approved allocations
no second accounting truth is introduced
Alembic revisions are split into reviewable chunks
slice 16 reporting introduces no base truth tables
the code structure still reads like Seeker, not like a separate subsystem
