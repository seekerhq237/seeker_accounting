2. Purpose of this blueprint

The purpose of this blueprint is to make sure Seeker payroll is built as a configuration-driven Cameroon payroll engine rather than a collection of fragile formulas hidden in code.

This is necessary because the current Cameroon payroll environment has multiple official/statutory engines that must be handled separately. The DGI publishes an official monthly salary-withholding barème for IRPP and related salary deductions excluding TDL, while TDL is handled separately. The published DGI table also shows the threshold break around 62,000 FCFA, where the IRPP/CAC lines drop to zero while other salary deductions continue.

CNPS also requires structured handling rather than one flat formula. Official CNPS material states that contributions are based on remuneration including cash and in-kind benefits, capped at 750,000 FCFA per month for PF and PVID, and that contribution floors depend on the minimum wage by sector. It also shows that family benefit rates vary by regime, and pension is split employer/employee at 4.2% + 4.2% = 8.4%. The employer is responsible for declaring and paying the whole contribution, including the employee share withheld from salary.

Your payroll handbook already points in the same direction: separate handling for IRPP, CAC, TDL, CNPS employee/employer portions, OHADA posting flow, payroll status workflow, and payroll-to-journal reconciliation.

3. Core design principle to lock

Seeker payroll for Cameroon must be:

table-driven
effective-dated
company-scoped
posting-controlled
journal-linked
audit-friendly

And must not be:

hard-coded around current rates
dependent on external payment APIs
dependent on bank/mobile-money integrations
implemented as a monolithic “gross-to-net” function with hidden rules

This aligns directly with Seeker’s locked schema principle that payroll rules live in tables, not just code, and with the product-wide rule that posted accounting truth lives in journals, not in UI tables or shortcut status fields.

4. Cameroon payroll compliance baseline for Seeker

For Seeker, Cameroon payroll should be modeled around these statutory and operational facts.

4.1 Separate payroll engines must exist

Seeker must support at least these distinct rule engines:

salary earnings engine
IRPP / related salary withholding barème engine
TDL engine
CNPS engine
employer levy / employer contribution engine
benefits-in-kind valuation engine
overtime engine

The reason is simple: the DGI withholding barème is published separately from TDL, and CNPS has separate branch/rate/cap logic.

4.2 Payroll must preserve separate bases

Seeker must explicitly preserve at least these payroll bases:

gross earnings
taxable salary base for DGI withholding
TDL base
CNPS contributory base
employer-cost base
net payable base

These should not be collapsed into one generic “taxable pay” field.

4.3 Payroll must support effective-dated rule changes

All statutory rules must be effective-dated because:

DGI tables can change
CNPS rates/caps/regime rules can change
company-specific payroll policy can change over time

This is already consistent with the existing Seeker payroll schema, which includes rule sets, brackets, and effective-dated compensation profiles.

4.4 Payroll payment tracking is in scope; payment execution is not

Seeker payroll must support:

payroll posted/unposted state
employee payment status
payment date
remittance tracking
liability settlement tracking

But must not support direct disbursement through external banking or mobile-money APIs. That is already a locked Seeker scope rule.

5. Seeker payroll conceptual model

The payroll domain should be locked conceptually like this:

Company Payroll Settings
→ Employee Master
→ Compensation Profile
→ Component Assignments / Variable Inputs
→ Rule Sets + Brackets
→ Payroll Run
→ Payroll Run Employees
→ Payroll Run Lines
→ Payslips / Summaries
→ Payroll Posting
→ Payment Tracking / Remittance Tracking

This gives Seeker four clean control layers:

First: setup truth

Who is paid, how they are categorized, and what statutory/company rules apply.

Second: calculation truth

How gross, deductions, employer cost, and net pay are derived.

Third: accounting truth

When payroll becomes journal-linked expense and liability truth.

Fourth: settlement truth

When net salary and statutory liabilities are tracked as paid or unpaid.

6.2 What should be added carefully

To make Cameroon payroll robust without drifting away, I recommend adding only a few focused support tables.

company_payroll_settings

One-to-one with company.

Purpose:

company payroll defaults
statutory pack version
CNPS regime
accident-risk class
default pay frequency
default payroll currency
overtime policy mode
benefit-in-kind policy mode
payroll numbering preferences

This table is the cleanest way to avoid stuffing company payroll logic into scattered services.

payroll_input_batches

Purpose:

approved variable inputs for a run period
overtime hours
bonuses
penalties
manual taxable/non-taxable adjustments
approved allowances

This avoids overloading compensation assignments with month-specific values.

payroll_payment_records

Purpose:

track actual internal payment status facts per payroll-run employee
partial/full payment support
payment reference
payment date
optional treasury linkage later

This is safer than relying only on a status flag.

payroll_remittance_batches

Purpose:

track DGI/CNPS/other remittances by payroll period/run
remittance date
amount due
amount paid
reference
status

This is the right accounting complement to payroll posting.

These are refinements, not a redesign.

8. Rule-set model to lock

payroll_rule_sets and payroll_rule_brackets should be the heart of statutory configurability.

Use them for:

DGI IRPP/related deduction barème
TDL table
pension contribution profiles
employer accident-risk classes

This is exactly the direction already described in the Seeker scope: Cameroon payroll should support PIT brackets, percentage-based deductions, fixed-scale taxes, employer accident-risk contributions, employee and employer pension contributions, and overtime premium logic through configuration tables rather than code rewrites.

9. Payroll accounting model to lock

Payroll accounting in Seeker should follow this model:

9.1 At payroll posting
debit salary/benefit/employer-cost expense accounts
credit employee net-pay liability
credit tax liabilities
credit CNPS and other statutory liabilities
9.2 At employee payment tracking
reduce payroll payable / net salary liability
track paid/unpaid/partial state internally
9.3 At statutory remittance tracking
reduce DGI/CNPS/other liability balances
track remittance status separately from payroll run status

This is consistent with your handbook’s OHADA posting flow and with Seeker’s global posting-link pattern, where posted source documents carry posted_journal_entry_id and create journal-linked truth through controlled posting services.
