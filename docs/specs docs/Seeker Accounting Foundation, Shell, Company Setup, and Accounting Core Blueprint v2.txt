# Seeker Accounting Foundation, Shell, Company Setup, and Accounting Core Blueprint v2

## 1. Purpose of this blueprint

This blueprint replaces the earlier first-slice blueprint and aligns it explicitly with the authoritative normalized schema.

Its purpose is to define the implementation-safe structure for the first build slices so that:

* folder and module names match the schema vocabulary
* services and repositories align with normalized table boundaries
* Codex does not invent vague model names that drift from the schema
* the foundation work can move cleanly into SQLAlchemy models and Alembic migrations

This document is now the authoritative bridge between:

* the clean scope
* the architecture blueprint
* the normalized database schema

## 2. What changed from the earlier blueprint

The earlier blueprint was directionally correct, but some names were too vague for safe implementation.

This corrected version fixes that by:

* replacing ambiguous names such as `tax_profile`, `account_group`, and `account_mapping`
* aligning file and model names to actual schema concepts such as `company_fiscal_defaults`, `tax_codes`, `account_classes`, `account_types`, and `account_role_mappings`
* making the first-slice package layout reflect the real phase-1 schema
* reserving proper module boundaries for administration and accounting-core reference data

This is important because Seeker Accounting is not a loose desktop CRUD tool. It is a multi-company accounting system whose workflow and reporting must reconcile back to accounting truth.

## 5. Dependency rules to lock

These rules are mandatory and unchanged in principle, but now expressed in schema-aligned terms.

### 5.1 Allowed dependency flow

* UI may depend on DTOs, services, shared UI helpers, shared enums, and validation presentation helpers.
* Services may depend on repositories, DTOs, shared validators, shared enums, shared utils, and platform services.
* Repositories may depend on database session or unit of work, ORM models, and repository-local mapping helpers.
* ORM models may depend only on SQLAlchemy base/types and carefully chosen shared enums.
* Platform services may be consumed by business services but must not reach upward into feature UI.

### 5.2 Forbidden dependency flow

* UI must not import repositories directly.
* UI must not open raw sessions and perform writes.
* Repositories must not import widget classes.
* Models must not import service classes.
* Accounting UI must not create journal rows directly.
* Seed workflows must not be hidden inside dialogs or pages.
* Feature packages must not bypass service contracts to mutate another module’s tables.

### 5.3 Practical meaning

In Access terms:

* forms are the surface
* services are the disciplined workflow layer
* repositories are the reusable data-access/query boundary
* posting services are the controlled replacement for ad hoc form-event accounting logic

## 6. Foundation package responsibilities

### 6.1 `main.py`

Responsibilities:

* application entry point only
* create Qt application
* call bootstrap pipeline
* start event loop

Must not contain:

* business logic
* database write logic
* shell composition logic beyond startup call

### 6.2 `app/bootstrap/application.py`

Responsibilities:

* orchestrate startup sequence
* load settings
* initialize logging
* initialize database engine and session factory
* initialize unit of work factory
* create service registry
* create shell window
* attach initial application context

### 6.3 `db/engine.py`

Responsibilities:

* create SQLAlchemy engine from config
* keep engine creation portable

### 6.4 `db/session.py`

Responsibilities:

* expose session factory
* avoid hidden global ad hoc sessions

### 6.5 `db/unit_of_work.py`

Responsibilities:

* define transaction boundary abstraction for write workflows
* keep commit and rollback outside widgets
* support service-level orchestration cleanly

### 6.6 `app/dependency/service_registry.py`

Responsibilities:

* centralize service construction and wiring
* expose service instances to pages/dialogs
* keep service creation out of UI widgets

## 7. Shell blueprint

### 7.1 Shell responsibilities

The shell hosts workflows. It does not own business rules.

It should provide:

* navigation frame
* workspace area
* active company display
* current user display
* fiscal context placeholder, then live fiscal context later
* command and action bar support where relevant
* notifications and error surfacing

### 7.2 Shell subcomponents

#### `main_window.py`

Owns:

* root frame
* top-level composition of sidebar, topbar, workspace host

Must not own:

* company persistence logic
* chart validation logic
* journal posting logic

#### `workspace_host.py`

Owns:

* page switching
* active workspace lifecycle

#### `sidebar.py`

Owns:

* module navigation buttons
* active-state presentation
* section grouping

#### `topbar.py`

Owns:

* active company label
* quick shell actions
* user menu placeholder
* fiscal context display later

### 7.3 Navigation IDs to lock now

* dashboard
* companies
* chart_of_accounts
* fiscal_periods
* journals
* reports
* administration

### 7.4 Shell behavior rules

* no giant scroll-heavy pages
* list-heavy modules should use lean action bars and strong tables/trees
* create/edit workflows should generally use dialogs unless a full workspace is clearly better
* active company must always be visible once selected
* if no company exists, the shell must degrade gracefully and lead the user into company setup

* `active_company_name` nullable

## 9. Administration foundation blueprint

This module must exist from the start even if its UI is initially thin.

### 9.1 Responsibilities

* user persistence foundation
* role and permission definitions
* role-permission assignments
* user-role assignments
* user-company access mappings
* access checks for active company selection and sensitive actions

### 9.2 Why it is in the first-stage structure

The normalized schema includes users, roles, permissions, user roles, role permissions, and user-company access in the phase-1 backbone. The product scope also explicitly requires users, roles, permissions, company access assignment, and controlled access to sensitive financial areas.

### 9.3 Early implementation stance

The first implementation pass may keep administration UI light, but the data models, repositories, and service boundaries must exist early because:

* company activation will depend on access checks later
* period control and journal posting will depend on permission enforcement
* audit trails need actor identity

## 10. Company module blueprint

### 10.1 Responsibilities

The company module covers the master company setup needed before real accounting activity begins.

Included in first pass:

* company listing
* create company
* edit company
* company preferences
* company fiscal defaults
* activate company into session context
* invoke company initialization and seed entry points

Not included in first pass:

* advanced branding assets
* branch hierarchy
* external integrations

### 10.2 Company persistence boundaries

These are distinct schema-aligned concepts and must remain distinct:

* `companies`
* `company_preferences`
* `company_fiscal_defaults`

Do not collapse these into one wide table or one vague repository.

### 10.3 Company service contracts

#### `CompanyService`

Primary responsibilities:

* create company
* update company
* list companies
* get company details
* update preferences and fiscal defaults through explicit workflows

Recommended methods:

* `list_companies() -> list[CompanyListItemDTO]`
* `get_company(company_id: int) -> CompanyDetailDTO`
* `create_company(command: CreateCompanyCommand) -> CompanyDetailDTO`
* `update_company(company_id: int, command: UpdateCompanyCommand) -> CompanyDetailDTO`
* `update_company_preferences(company_id: int, command: UpdateCompanyPreferencesCommand) -> CompanyPreferencesDTO`
* `update_company_fiscal_defaults(company_id: int, command: UpdateCompanyFiscalDefaultsCommand) -> CompanyFiscalDefaultsDTO`
* `deactivate_company(company_id: int) -> None`

Validation rules:

* legal name required
* display name required
* base currency required
* country required
* uniqueness rules enforced consistently

#### `CompanyContextService`

Recommended methods:

* `set_active_company(company_id: int, user_id: int | None = None) -> ActiveCompanyDTO`
* `get_active_company() -> ActiveCompanyDTO | None`
* `clear_active_company() -> None`

Important rule:
This service must later validate user-company access, not merely set runtime state.

#### `CompanySeedService`

Primary responsibilities:

* initialize baseline company setup after creation
* trigger OHADA chart seed
* trigger starter tax code seed
* create default document-sequence definitions where applicable

Recommended methods:

* `initialize_new_company(company_id: int, template_code: str | None = None) -> CompanyInitializationResultDTO`
* `seed_ohada_chart(company_id: int) -> SeedResultDTO`
* `seed_default_tax_codes(company_id: int) -> SeedResultDTO`
* `seed_document_sequences(company_id: int) -> SeedResultDTO`

Important rule:
Seeding belongs in service workflows, not in create-dialog click handlers.

### 10.4 Company repository contracts

#### `CompanyRepository`

Responsibilities:

* list company records
* fetch by id
* fetch by legal/display name for validation
* persist company records

#### `CompanyPreferenceRepository`

Responsibilities:

* get company preferences
* create or update preferences

#### `CompanyFiscalDefaultRepository`

Responsibilities:

* get fiscal defaults
* create or update fiscal defaults

## 11. Accounting reference-data blueprint

This is the area that needed the biggest correction from the earlier blueprint.

### 11.1 Responsibilities

This package exists to hold normalized accounting reference/configuration concepts that the chart, periods, journals, and later operational modules depend on.

Included in early scope:

* currencies
* countries
* document sequences
* payment terms
* tax codes
* tax code account mappings
* account classes
* account types
* account role mappings

### 11.2 Why this package matters

Without this package, Codex is likely to scatter tax setup, numbering setup, and account-role assumptions into the company module or chart module in vague ways. That would not fit the normalized schema.

### 11.3 Key service contracts

#### `ReferenceDataService`

Responsibilities:

* list and expose reusable reference data for early modules
* support UI dropdowns and selection datasets cleanly

#### `TaxSetupService`

Responsibilities:

* create/update/list tax codes
* manage tax code account mappings
* ensure company-scoped tax configuration remains explicit

Recommended methods:

* `list_tax_codes(company_id: int) -> list[TaxCodeDTO]`
* `create_tax_code(company_id: int, command: CreateTaxCodeCommand) -> TaxCodeDTO`
* `update_tax_code(company_id: int, tax_code_id: int, command: UpdateTaxCodeCommand) -> TaxCodeDTO`
* `set_tax_code_account_mapping(company_id: int, command: SetTaxCodeAccountMappingCommand) -> TaxCodeAccountMappingDTO`

#### `NumberingSetupService`

Responsibilities:

* manage company-scoped document-sequence definitions
* expose next-number preview behavior later where needed

#### `AccountRoleMappingService`

Responsibilities:

* assign and validate functional role mappings such as AR control, AP control, inventory control, and payroll payable

Important rule:
Do not create one vague `account_mapping` model or service. Use explicit normalized mapping concepts.

## 12. Chart of accounts blueprint

### 12.1 Responsibilities

This remains a backbone accounting module.

Included in first chart pass:

* company-scoped chart of accounts
* account hierarchy
* create account
* edit account
* deactivate account
* account search/filter
* OHADA chart seed integration

### 12.2 Schema-aligned account model direction

The `accounts` persistence model should align to these normalized concepts:

* `company_id`
* `account_code`
* `account_name`
* `account_class_id`
* `account_type_id`
* `parent_account_id`
* `normal_balance`
* `allow_manual_posting`
* `is_control_account`
* `is_active`
* `notes`

Important rule:
Do not model `account_class` or `account_type` as casual free text inside the account UI/service layer. They map to normalized reference entities.

### 12.3 Account design rules

* account code must be unique per company
* parent-child hierarchy must not allow cycles
* parent account must belong to same company
* inactive accounts remain part of historical reporting context
* control-account designation is explicit
* allow-manual-posting is separate from active status

### 12.4 Chart service contracts

#### `ChartOfAccountsService`

Responsibilities:

* list accounts by company
* return tree and flat views
* create/update/deactivate accounts
* validate code uniqueness and hierarchy rules

Recommended methods:

* `list_accounts(company_id: int, filters: ChartFiltersDTO | None = None) -> list[AccountListItemDTO]`
* `get_account(company_id: int, account_id: int) -> AccountDetailDTO`
* `create_account(company_id: int, command: CreateAccountCommand) -> AccountDetailDTO`
* `update_account(company_id: int, account_id: int, command: UpdateAccountCommand) -> AccountDetailDTO`
* `deactivate_account(company_id: int, account_id: int) -> None`
* `get_account_tree(company_id: int) -> list[AccountTreeNodeDTO]`
* `validate_account_code(company_id: int, proposed_code: str, exclude_account_id: int | None = None) -> ValidationResultDTO`

#### `ChartSeedService`

Responsibilities:

* load deterministic OHADA starter chart for a company
* guard against accidental duplicate seed workflows

Recommended methods:

* `seed_default_chart(company_id: int, template_code: str = "OHADA_DEFAULT") -> SeedResultDTO`
* `preview_chart_seed(template_code: str = "OHADA_DEFAULT") -> ChartSeedPreviewDTO`

Important rule:
Seeding must be versionable and deterministic enough to support migration, support, and repeatable setup.

### 12.5 Chart repository contract

#### `AccountRepository`

Responsibilities:

* fetch account by company and id
* fetch by company and account code
* list by company
* provide tree-source data
* persist account records

Avoid:

* unscoped account queries
* business-rule enforcement in repository code

## 19. What Codex must not do in these slices

Codex must not:

* collapse repositories and services into one convenience file
* put SQLAlchemy session logic directly into widgets
* invent vague models such as `tax_profile` or `account_group`
* replace normalized references with loose free-text fields where the schema defines separate entities
* make chart of accounts a flat unstructured CRUD grid
* hide seed workflows inside UI button handlers
* redesign unrelated future modules while implementing these slices
* bypass company scoping in repositories

## 20. Acceptance checkpoints for this corrected blueprint

### Foundation is acceptable when:

* app boots through a clean bootstrap pipeline
* service wiring is centralized
* database/session/unit-of-work setup is isolated and portable

### Shell is acceptable when:

* it hosts pages cleanly
* active company is visible
* page switching is stable
* dialogs and notifications have consistent patterns

### Administration foundation is acceptable when:

* user/role/permission/company-access models and service boundaries exist
* company activation can later validate access without redesign

### Company setup is acceptable when:

* companies can be listed, created, edited, and selected
* preferences and fiscal defaults are stored through distinct schema-aligned boundaries
* initialization and seeding are service-driven

### Accounting reference data is acceptable when:

* tax codes, numbering setup, account classes/types, and account-role mappings are represented explicitly
* no vague mapping models exist

### Chart of accounts is acceptable when:

* accounts are company-scoped
* hierarchy works
* duplicate codes are blocked per company
* create/edit/deactivate flows are service-driven
* default OHADA chart can be seeded safely

### Fiscal periods and journals are acceptable when:

* periods support open/close/lock semantics
* journal entries and lines are modeled correctly
* posting is controlled and balancing is enforced

## 21. What this means in Access terms

This corrected blueprint is the equivalent of deciding, before implementation:

* which forms are only UI surfaces
* which business routines own validation and posting control
* which query/data routines belong in structured repositories
* how company context flows through the whole application
* how chart setup, tax setup, period control, and journals stay normalized instead of becoming loose setup forms

That discipline is what keeps Seeker Accounting from becoming a prettier but fragile Access-style monolith.
