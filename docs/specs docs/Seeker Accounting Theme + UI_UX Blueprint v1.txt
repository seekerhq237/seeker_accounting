Seeker Accounting Theme + UI/UX Blueprint v1
1. Product feel to lock

Seeker Accounting should feel like this:

Calm, premium, trusted, operator-focused.

Not flashy. Not playful. Not generic Qt.
Think:

QuickBooks’ task clarity
a cleaner, denser desktop workflow
stronger accounting seriousness
better dark theme discipline
less clutter, less wasted space, fewer oversized cards

The emotional target is:
“This looks modern, expensive, and safe enough to run a real business.”

2. What we borrow from QuickBooks, and what we do not

We borrow:

job-centered left navigation
always-available create actions
dashboard views around operational priorities
friendly but professional visual tone
clean status chips and inline summaries
report drilldown mindset
low-friction movement between overview and detail

We do not borrow:

web-app looseness
oversized empty space
over-carded layouts
shallow dashboards with too many widgets
soft contrast that hurts dense accounting work
consumer-style playfulness in financial screens

Also, QuickBooks’ current web experience appears light-first in some environments, while your product explicitly wants both light and dark. So Seeker should treat dual-theme support as a product decision of its own, not as a literal copy of QBO behavior.

3. Brand direction from your logo

Your logo gives us the right base:

Primary brand blue: #2363EA
White as the clean contrast color
Black wordmark in the supplied asset, but in-app wordmark should adapt by theme

That means the UI should use blue as an accent, not as a flood color.
The app should mostly be built from refined neutrals, with the blue reserved for:

primary actions
active navigation
focus rings
selected rows
key charts
important interactive highlights

That is the right professional move. Financial software needs trust first, branding second.

5. Theme system

We should build the theme as a tokenized design system, not loose stylesheet hacking.

Theme layers:

semantic color tokens
typography tokens
spacing scale
radius scale
elevation/shadow scale
component state rules
icon style rules

Files should later map into your reserved styles area under shared/ui/styles/ and shell widgets, not page-by-page overrides.

7. Dark theme

This should not be a lazy color inversion.

Nielsen Norman Group notes that dark mode is popular, but it needs careful testing because contrast, glow, and readability failures are common. WCAG also still applies: body text should maintain at least 4.5:1 contrast, and non-text UI components should maintain at least 3:1 where required.

7.2 Dark theme rules
never use pure black cards on pure black background
never use thin gray text
reduce shadow, increase border separation
selected rows should use tinted fill, not neon outline
forms need stronger field boundaries than in light mode
charts need fewer colors and clearer labels
8. Typography

Typography should feel premium and accounting-friendly.

Recommended hierarchy:

App title / module title: 22–24 px, semibold
Section title: 16–18 px, semibold
Card title / group title: 14–15 px, semibold
Body: 13–14 px
Dense table text: 12.5–13 px
Caption / meta text: 11–12 px

Rules:

prefer semibold over bold everywhere
avoid giant headers
numeric values align right in tables
financial totals use tabular numerals if the chosen font supports them
labels must remain readable in both themes
9. Density and spacing

This is desktop accounting software. It should be comfortable, but dense enough for work.

Material’s density guidance explicitly notes that greater density is useful for lists, tables, and long forms. NN/g also stresses that tables must support finding, comparing, viewing/editing, and acting on records.

So the Seeker density rules should be:

page outer padding: 20 px
content group spacing: 16 px
control gap inside bars: 8–10 px
field height: 34–38 px
action button height: 32–36 px
table row height default: 34 px
dense row option later: 30 px

No giant tiles by default.
No oversized buttons.
No bloated headers.

10.5 Tables

This is one of the most important parts.

NN/g identifies four major table tasks: find records, compare them, view/edit/add a row, and take actions. Seeker tables must support all four.

So every major table should support:

sticky header
quick search
filters
visible sort state
row selection
bulk actions where useful
double-click open
context menu
export
totals footer where relevant
frozen key columns in some modules later

Accounting tables must look premium, not developer-default.

11. Form UX rules

NN/g’s recent form guidance emphasizes structure, transparency, clarity, and support to reduce cognitive load. That is exactly the right rule here.

So Seeker forms should follow this:

group fields by business meaning
show only the fields needed for the current task
use progressive disclosure for advanced settings
give helper text only where confusion is likely
validate early, but not aggressively
do not show giant stacked forms on every main page
use dialogs for create/edit unless the workflow truly needs a full workspace

That already matches your shell blueprint.

13. Module page blueprint

Most modules should follow one of only three page patterns.

Pattern A: List workspace

For customers, suppliers, chart of accounts, employees, assets.

Structure:

compact title/context row
action bar
filters/search
main table/tree
bottom status or totals strip if needed
Pattern B: Document workspace

For invoices, bills, receipts, payments, journals.

Structure:

document header
status/action row
line grid
totals panel
side summary or posting panel
Pattern C: Report workspace

For trial balance, ledger, income statement, balance sheet, aging.

Structure:

report filter bar
report canvas
drilldown interaction
export/print actions
comparison controls

QuickBooks’ current reporting direction is moving toward a modern view with features like auto-refresh and drilldowns; Seeker should adopt that mindset, but in a desktop-first way.

14. Accessibility and readability rules to lock

These are non-negotiable:

normal text target: minimum 4.5:1 contrast
large text target: minimum 3:1
non-text UI controls and boundaries: minimum 3:1 where applicable
color is never the only status signal
keyboard navigation works across tables, forms, dialogs
focus states must be visible in both themes
error messages must be plain-language and local to the workflow

WCAG supports these thresholds directly.

15. Motion and feedback

Motion should be restrained.

Allowed:

subtle hover transitions
dialog fade/scale
selection transitions
toast slide-in
progress indicators

Avoid:

flashy animation
bouncing widgets
dashboard gimmicks

Finance software should feel responsive, not theatrical.

16. Seeker-specific UX principles to lock
Accounting context is always visible.
Company and fiscal context stay in view.
Posting is visually serious.
Post actions are distinct from save draft.
Dense work beats decorative layout.
The operator should see more records, not more whitespace.
Tables are first-class citizens.
This is accounting software, not a marketing site.
Dialogs reduce page bloat.
Main pages stay focused.
Status must be instantly legible.
Draft, posted, locked, overdue, partial, paid.
Theme consistency beats individual screen creativity.
We do not reinvent components page by page.
