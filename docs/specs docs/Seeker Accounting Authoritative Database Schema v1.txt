# Seeker Accounting Authoritative Database Schema v1

## 1. Purpose

This document defines the authoritative relational database schema direction for Seeker Accounting.

It is designed to satisfy the locked product scope and architecture:

* multi-company support with strict data separation
* OHADA-ready accounting foundation
* double-entry integrity as the system backbone
* draft versus posted workflow separation
* sales/AR, purchases/AP, cash and bank, inventory, fixed assets, payroll, reporting, and administration
* SQLite first, but portable to PostgreSQL, Firebird, and similar relational engines
* clean modular-monolith application architecture with database logic isolated behind repositories and services

This schema is intentionally normalized and conservative. The goal is not to optimize for shortcuts or raw data-entry convenience. The goal is to create a stable accounting data backbone that can grow cleanly.

---

## 2. Core schema design principles

### 2.1 One application database, company-scoped business data

Version 1 will use one application database. Business tables are company-scoped unless they are truly global platform tables.

Implication:

* almost every business table has `company_id`
* uniqueness is usually composite with `company_id`
* repositories and services must always scope reads and writes by company

### 2.2 Accounting truth is centralized

Operational documents do not become accounting truth by themselves.

The accounting truth lives in:

* `journal_entries`
* `journal_entry_lines`

Operational documents such as invoices, bills, treasury transactions, depreciation runs, and payroll runs create accounting impact through controlled posting workflows and store linkage back to the resulting journal entry where applicable.

### 2.3 No stored running balances in base tables

The base schema should not store values such as:

* customer current balance
  n- supplier current balance
* account running balance
* stock balance on hand

Those are derived from posted transactions, allocations, and stock movements.

Why:

* prevents update anomalies
* preserves normalization
* keeps reporting tied to accounting truth

Where performance later needs help, we can add indexed summary queries, cached reporting tables, or controlled refresh tables. Those are optimization artifacts, not the base truth model.

### 2.4 Header/line structure for business documents

Documents that naturally contain multiple rows are split into header and line tables.

Examples:

* sales invoice / sales invoice lines
* purchase bill / purchase bill lines
* manual journal / journal lines
* inventory document / inventory document lines
* payroll run / payroll run lines

### 2.5 Allocations are separate facts

Settlement is not stored by mutating source documents alone.

Examples:

* customer receipt applied to invoice = allocation table
* supplier payment applied to bill = allocation table
* bank reconciliation match = reconciliation match table

This is one of the most important normalization choices in the whole schema.

### 2.6 Reference tables where they add clarity and portability

Use reference tables for meaningful business concepts that are reused or configurable.

Examples:

* currencies
* payment terms
* tax codes
* payroll rule sets and brackets
* document sequence definitions

Do not over-normalize trivial UI-only lists that can safely live as application enums.

### 2.7 Portable primary keys and constraints

Use conventional integer surrogate keys for most tables.

Rules:

* primary key: `id`
* foreign keys use `*_id`
* business codes use separate unique or composite unique constraints
* avoid engine-specific generated columns or vendor-only types in base design

### 2.8 Soft deactivation over destructive deletion

For most master tables:

* use `is_active`
* optionally `deactivated_at`
* do not hard-delete records that may already be referenced by posted history

Examples:

* accounts
* customers
* suppliers
* employees
* financial accounts
* items

### 2.9 Effective dating where legal or payroll logic changes over time

Payroll and tax configuration should support change over time without code rewrites.

Therefore use effective dates in:

* tax rates
* payroll rule sets
* employee compensation assignments
* contribution brackets where applicable

## 3. Naming and common-column conventions

### 3.1 Table naming

Use plural snake_case table names.

Examples:

* `companies`
* `accounts`
* `journal_entries`
* `sales_invoices`

### 3.2 Standard metadata columns

Most business tables should include these unless there is a strong reason not to:

* `created_at`
* `updated_at`
* `created_by_user_id` nullable initially if system-generated
* `updated_by_user_id` nullable

Posted documents often also need:

* `posted_at`
* `posted_by_user_id`

### 3.3 Common status approach

Do not create separate status tables for every small workflow status unless they are truly configurable.

For v1, use controlled string codes or application enums persisted in columns like:

* `status_code`
* `document_status`
* `payment_status`

This is a practical balance between normalization and maintainability.

### 3.4 Global versus company-scoped tables

Global tables:

* users
* roles
* permissions
* role_permissions
* user_roles
* currencies
* countries

Company-scoped tables:

* companies
* fiscal_years
* accounts
* customers
* suppliers
* items
* employees
* journals
* invoices
* bills
* payroll runs
* tax codes
* financial accounts
* document sequences

Purpose: many-to-many mapping between roles and permissions.

Notes:

* payloads should be stored carefully and not abused as a replacement for real relational design

## 7.7 `journal_entries`

Purpose: accounting journal header.

Columns:

* `id` PK
* `company_id` FK -> companies.id
* `entry_number`
* `journal_type_code`
* `entry_date`
* `fiscal_period_id` FK -> fiscal_periods.id
* `description`
* `source_module_code` nullable
* `source_document_number` nullable
* `status_code`
  expected values: draft, posted, reversed
* `posted_at` nullable
* `posted_by_user_id` nullable FK -> users.id
* `reversal_of_journal_entry_id` nullable FK -> journal_entries.id
* `created_at`
* `updated_at`
* `created_by_user_id` nullable FK -> users.id
* `updated_by_user_id` nullable FK -> users.id

Constraints:

* unique (`company_id`, `entry_number`)

Notes:

* this is the accounting truth header
* do not try to store debit and credit totals as permanent truth if avoidable; if stored for convenience, they must be system-maintained and always recomputable from lines

## 7.8 `journal_entry_lines`

Purpose: accounting line items.

Columns:

* `id` PK
* `journal_entry_id` FK -> journal_entries.id
* `line_number`
* `account_id` FK -> accounts.id
* `description` nullable
* `debit_amount`
* `credit_amount`
* `party_type_code` nullable
  customer, supplier, employee, none
* `party_id` nullable
* `tax_code_id` nullable FK -> tax_codes.id
* `currency_code` nullable FK -> currencies.code
* `exchange_rate` nullable
* `source_line_reference` nullable
* `created_at`

Constraints:

* unique (`journal_entry_id`, `line_number`)
* check-like rule in service/app logic: one of debit or credit must be positive and the other zero

Notes:

* `party_type_code` + `party_id` is a controlled polymorphic reference for subledger reporting. This is one of the few places where a typed reference is acceptable because journal lines may legitimately target multiple party domains.
* stricter integrity is still achieved because source operational documents keep their own normal foreign keys.

## 16. Normalization review by area

### 16.1 First normal form

Achieved because:

* no repeating groups inside tables
* header/line patterns are separated correctly
* multiple allocations are represented as separate rows

### 16.2 Second normal form

Achieved because:

* non-key attributes in line tables depend on the whole row identity, not only part of it
* many-to-many mappings are split into dedicated tables such as `role_permissions`, `user_roles`, `customer_receipt_allocations`, and `supplier_payment_allocations`

### 16.3 Third normal form

Achieved because:

* party balances are not stored redundantly in customer/supplier tables
* stock balances are not stored as base truth on items
* account running balances are not stored on the account master
* payroll rules are separated from payroll runs
* tax codes and account mappings are separated from invoices and bills

### 16.4 Deliberate controlled exceptions

Some values are stored even though they are derived, because they represent document or posting snapshots that must remain frozen for audit and reporting consistency.

Examples:

* invoice subtotal/tax/total
* bill subtotal/tax/total
* payroll run employee gross/net values
* depreciation run outputs

These are acceptable because they are document-state facts controlled by service logic, not casual denormalized convenience columns.

---

## 17. Key relationship patterns to protect

### 17.1 Company scoping

For almost every business relation:

* child rows must belong to the same company as their parent workflow
* services must validate same-company ownership

### 17.2 Posting link pattern

Documents that create accounting impact should usually carry `posted_journal_entry_id`.

Examples:

* sales_invoices
* customer_receipts
* purchase_bills
* supplier_payments
* treasury_transactions
* treasury_transfers
* inventory_documents
* asset_depreciation_runs
* payroll_runs

This keeps posting traceable and simple.

### 17.3 Allocation pattern

Settlement never mutates source totals as truth.

Use:

* `customer_receipt_allocations`
* `supplier_payment_allocations`
* `bank_reconciliation_matches`

### 17.4 Control-account pattern

Subledger modules do not directly replace GL control.

Use:

* `account_role_mappings`
* journal posting services
* subledger reports built from source docs + posted journals

## 19. What not to do in this schema

Do not:

* store customer balance on `customers`
* store supplier balance on `suppliers`
* store stock on hand on `items` as the base truth
* store GL balance on `accounts`
* let UI tables directly become accounting truth
* let operational modules bypass journals when posted
* mix multiple business concepts in one giant “transactions” table
* use SQLite-only hacks as design foundations
* overuse free-form JSON columns for structured accounting logic

## 21. Access analogy for clarity

In Access terms, this schema deliberately prevents the classic long-term problem where:

* forms write directly into disconnected tables
* balances get stored in many places
* a payment “marks” an invoice paid with a yes/no field
* inventory quantity is just edited manually on the item record
* payroll formulas are hidden in form code

Instead:

* headers and lines are separated cleanly
* allocations are their own facts
* posting creates ledger truth
* period control has its own tables
* company separation is structural
* payroll rules live in tables, not just code

That is the difference between a product-grade accounting database and an overgrown forms database.

---

## 22. Schema decisions explicitly locked by this document

1. One application database with company-scoped business tables.
2. General ledger tables are the accounting truth.
3. No base-table running balances for AR, AP, stock, or GL.
4. Allocation tables are mandatory for settlement workflows.
5. Header/line decomposition is mandatory for document workflows.
6. Tax, payroll, numbering, and account-role rules are table-driven.
7. Inventory and payroll are modeled as first-class modules, not as add-ons.
8. The schema remains portable and avoids engine-specific dependencies.
9. Soft deactivation is preferred over destructive deletion for master data.
