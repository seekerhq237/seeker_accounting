Authoritative Database Schema Extension
Projects, Contracts, Job Costing, Budgeting, and Management Accounting v1
1. What we are locking

We are adding a new schema family for:

contracts
projects
jobs / work packages
cost codes
budget versions and budget lines
commitments
project-aware transaction dimensions
payroll-to-project allocation bridge

This extension is management-accounting structure layered over the existing accounting spine.

It does not create a second ledger.
It does not replace source modules.
It does not introduce one giant generic “project transactions” table.

That last point is especially important because the current Seeker schema already explicitly rejects letting operational modules bypass journals or mixing multiple business concepts into one vague transaction table.

2. Updated table-group map

The existing schema has 11 table groups. We should extend it like this:

Platform and administration
Company and system setup
Fiscal periods and accounting core
Customers, suppliers, and common commercial masters
Sales and accounts receivable
Purchases and accounts payable
Treasury, bank, and reconciliation
Inventory and stock valuation
Fixed assets
Payroll
Projects, contracts, job costing, budgeting, and management accounting
Import, export, and operational support

That keeps the new domain explicit and prevents it from being scattered across unrelated modules. This is consistent with the current feature-based modular-monolith structure and the way the existing schema is grouped.

This is the cleanest way to integrate payroll without redesigning payroll around the project module.

6. Existing-table extensions

The current schema already treats source documents as their own modules and uses posting links back to journals. This extension should preserve that by adding project dimensions to source tables rather than inventing detached project documents.

No structural change needed.

7. Integrity rules to lock

These are the rules that matter most.

7.1 Company-scoping rules
every new business header/master table is company-scoped
all cross-table references must resolve within the same company
repository and service methods remain company-aware

This follows the current Seeker isolation model directly.

7.2 Dimension-consistency rules
project_job_id requires a matching project_id
project_cost_code_id may exist without job, but not without at least project or contract context where the module requires it
if contract_id and project_id are both present, they must be compatible
job must belong to project
project linked to contract must belong to same company and, where applicable, same customer
7.3 Workflow-state rules
contracts and projects are operational masters, not posting records
budget versions and commitments are management-control workflows, not posting workflows
source modules continue to own posted accounting impact
posted source documents remain the basis for actual financial truth
7.4 No duplicate truth rules

Do not store as permanent base truth:

project actual cost balance
project actual revenue balance
project remaining budget
current contract value
project margin
commitment-consumed amount

Those are derived from approved budgets, commitments, and posted source transactions.

If performance later needs help, add controlled reporting caches or refresh tables, exactly the way the current schema already allows for optimization artifacts without replacing base truth.

8. What is allowed to be stored

These are acceptable stored snapshots:

project_budget_versions.total_budget_amount
document header totals already existing in source documents
project_commitments.total_amount
allocated_cost_amount on payroll allocation bridge
approved change-order deltas

Why acceptable:
they are document-state facts under service control, not casual denormalized balances. That matches the logic already accepted in the main Seeker schema for invoice totals, bill totals, payroll outputs, and depreciation-run outputs.

9. What we are deliberately not adding now

Not in this first extension pass:

project time-entry / timesheet engine
purchase orders
commitment-to-actual exact consumption link table
WIP posting tables
earned-value management tables
revenue-recognition engine tables
asset multi-project allocation tables
project transaction cache tables

Those can come later, but they should not enter the base extension until the first project-costing layer is stable.

11. Migration sequence for this extension

This is the safest order.

Phase A: setup extension
company_project_preferences
add new document-sequence type codes through seed/setup service
reserve new account role codes
Phase B: project and contract masters
contracts
contract_change_orders
projects
project_jobs
project_cost_codes
Phase C: budget and commitment structure
project_budget_versions
project_budget_lines
project_commitments
project_commitment_lines
Phase D: source-document integration
extend sales, purchases, treasury, inventory, and journal lines with project dimensions
Phase E: payroll bridge
payroll_run_employee_project_allocations
Phase F: reporting/query layer

No new truth tables.
Build service/query models for:

contract value
billed-to-date
actual cost
commitment exposure
budget vs actual
budget vs commitment
project profitability
contract profitability

This sequencing respects the current Seeker rule that dependent modules should be layered only after the accounting and operational spine is in place.

12. Decisions explicitly locked
Contracts, projects, jobs, budgets, and commitments are first-class schema concepts.
GL remains the financial truth.
No second ledger and no giant generic project-transaction table.
Actual cost comes from posted source documents and controlled journals.
Budgets and commitments are management-control facts, not ledger facts.
Project dimensions are added to source lines, not bolted on in UI-only state.
Payroll integrates through an allocation bridge, not by redesigning payroll tables into project tables.
Company scoping remains mandatory throughout.
Current contract value and project profitability are derived, not stored as base truth.
We stay portable and avoid SQLite-specific tricks.
